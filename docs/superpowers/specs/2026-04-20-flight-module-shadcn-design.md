# 飞行模块 `shadcn-vue` 改造设计

- 日期：2026-04-20
- 作用范围：`dev` 分支飞行模块
- 第一阶段目标页：`航线管理`
- 后续复用页：`任务管理`、`飞行记录`

## 1. 背景

当前项目是一个已经上线开发的 Vue 3 管理系统，核心技术栈包括：

- Vue 3
- Vite
- UnoCSS
- Element Plus
- vxe-table

当前仓库里已经有不少业务逻辑和页面结构，不适合为了接入 `shadcn` 风格而重建整个项目。

用户已经明确表达了以下偏好：

- 不要求与 `WaylineUI` HTML 原型一比一还原
- 更希望页面整体风格靠近 `shadcn-vue` 的默认视觉效果
- 希望先从飞行模块开始
- 希望先把已有页面重做，再逐步补空白页

## 2. 现状判断

### 2.1 已完成程度较高的飞行模块页面

- `src/views/route/index.vue`：航线管理
- `src/views/flight/task/index.vue`：任务管理
- `src/views/flight/record/index.vue`：飞行记录

这些页面已经具备较完整的业务逻辑和交互，但 UI 主要依赖 Element Plus 和项目自定义 SCSS。

### 2.2 明确未完成的飞行模块页面

- `src/views/flight/dispatch/index.vue`：一键调飞
- `src/views/flight/console/index.vue`：飞行控制台

这两页当前基本只有标题和说明文字。

### 2.3 原型资源现状

`WaylineUI` 文件夹中已经存在以下和飞行模块高度相关的原型：

- `任务管理.html`
- `航线管理.html`
- `一键调飞.html`
- `飞行控制台.html`
- `飞行记录.html`
- `查看-飞行记录.html`
- `新增航线-点状.html`
- `新增航线-环状.html`
- `新增航线-面状.html`

这些原型更适合当作“信息结构和内容参考”，而不是直接复制为生产代码。

## 3. 本次设计的核心目标

本次不是简单“换皮肤”，而是建立一套可以长期复用的页面体系。

### 3.1 目标

- 在当前 Vue 项目中安全接入 `shadcn-vue`
- 保留现有飞行模块业务逻辑
- 先把 `航线管理` 做成新的样板页
- 形成可复制的方法，后续迁移到 `任务管理`、`飞行记录`
- 页面视觉更接近 `shadcn-vue` 默认审美，而不是传统后台 UI

### 3.2 非目标

- 第一阶段不删除所有 Element Plus 依赖
- 第一阶段不重建整个项目样式体系
- 第一阶段不深度重写地图绘制、轨迹播放等复杂能力
- 第一阶段不强求和 HTML 原型逐像素一致

## 4. 方案对比

### 方案 A：整页一次性推倒重做

做法：

- `航线管理` 整页一次性改成全新 `shadcn-vue` 组件体系
- 交互、弹层、表单、地图区域同步重写

优点：

- 页面风格变化最快
- 视觉结果最完整

缺点：

- 代码改动太大
- 容易破坏现有业务逻辑
- 首次落地风险高

### 方案 B：桥接式改造

做法：

- 先把 `shadcn-vue` 基础层接入现有项目
- 先重做页面骨架、筛选区、统计区、卡片区、弹层外壳
- 复杂业务能力分阶段迁移

优点：

- 风险最低
- 可以快速做出样板页
- 适合后续复用到其他页面

缺点：

- 过渡阶段会有新旧组件共存

### 方案 C：只做视觉拟态

做法：

- 仍以 Element Plus 为主
- 只调整颜色、边框、圆角和排版，做出类似 `shadcn` 的效果

优点：

- 开发速度快

缺点：

- 只是“看起来像”
- 后面要真正迁移时还要再做一轮

### 结论

选择 **方案 B：桥接式改造**。

原因：

- 它最适合当前“已有业务逻辑较多”的项目
- 它能在风险可控的前提下，真正建立新的页面与组件体系
- 它更符合后续持续迁移飞行模块其它页面的需求

## 5. 技术接入设计

### 5.1 接入原则

- 当前项目继续保留 `Vue 3 + Vite + UnoCSS`
- 不使用 `shadcn/create` 新建项目模板
- 不重建整个工程
- 直接在当前项目中接入 `shadcn-vue`

原因：

- 当前仓库不是 React 项目
- 用户想改的是现有 Vue 工程
- `shadcn-vue` 更适合当前技术栈

### 5.2 第一阶段共存策略

- `Element Plus` 暂时保留
- `shadcn-vue` 负责新页面的主视觉和新组件层
- 旧页面继续可运行
- 新旧两套体系短期共存

### 5.3 样式策略

当前项目已有大量全局 SCSS 和基于 Element Plus 变量的样式。

第一阶段不继续把新页面的大量样式堆进现有 `common.scss`，而是采用：

- 新增基础 token 和 `shadcn-vue` 所需样式入口
- 新公共组件尽量自包含
- 页面样式按“页面内聚 + 可抽公共块”的方式组织

这样可以避免后续样式文件越来越重，也便于迁移剩余页面。

### 5.4 第一批基础组件

第一阶段需要的基础组件以 `shadcn-vue` 常用组件为主：

- Button
- Card
- Badge
- Input
- Select
- Dialog
- Sheet
- Separator
- Skeleton
- Pagination

说明：

- 如果某些功能型组件一时没有合适替代，可以临时保留旧实现
- 但页面外层结构、视觉骨架、交互入口应优先使用新的组件体系

## 6. `航线管理` 页面设计

### 6.1 页面定位

`航线管理` 是飞行模块第一张样板页，用来验证：

- 新视觉是否成立
- 新组件结构是否可复用
- 现有业务逻辑是否能平稳迁移

### 6.2 页面目标

用户进入页面后，需要快速完成三件事：

1. 看清航线总体情况
2. 通过筛选快速找到目标航线
3. 对单条航线做查看、编辑、删除等操作

### 6.3 页面结构

#### 区域一：页面头部

包含：

- 面包屑
- 页面标题
- 简短说明
- 主操作按钮（新增航线）

设计原则：

- 用 `shadcn-vue` 默认风格的轻量头部
- 不再沿用当前重装饰的大块英雄区域

#### 区域二：统计卡片区

包含四张卡：

- 航线总量
- 点状航线
- 面状航线
- 环状航线

设计原则：

- 视觉轻、信息清晰
- 强调数字和状态，不靠重色块取胜

#### 区域三：筛选区

保留现有筛选逻辑：

- 航线名称
- 航线类型
- 创建人
- 创建时间
- 更新时间
- 查询/重置

设计原则：

- 结构清爽
- 更像现代产品控制栏，而不是传统后台密集表单

#### 区域四：航线卡片列表区

保留当前“卡片化展示”的方向。

单张卡片重点展示：

- 航线名称
- 航线类型
- 当前状态
- 机型、创建人、时间等元信息
- 关键统计信息
- 操作按钮

设计原则：

- 卡片更轻
- 层次更清楚
- 用 `Card + Badge + Button + Separator` 组合
- 避免过度装饰和边框堆砌

#### 区域五：新增/编辑航线弹层

建议使用：

- `Sheet` 作为优先方案
- `Dialog` 作为保底方案

第一阶段保留：

- 现有表单逻辑
- 核心提交流程

第一阶段不追求：

- 完整重写地图规划交互
- 一次重做所有点状/环状/面状绘制体验

#### 区域六：空状态与分页

- 分页保留
- 空状态统一换成新风格
- 加载中使用骨架屏或轻量 loading

## 8. 实施顺序

第一阶段按以下顺序推进：

1. 接入 `shadcn-vue`
2. 建立基础 UI 组件和样式入口
3. 重做 `航线管理` 页面的头部、统计区、筛选区
4. 重做航线卡片列表
5. 重做新增/编辑弹层外壳
6. 验证业务逻辑无损

第二阶段：

- 复用方法改造 `任务管理`

第三阶段：

- 复用方法改造 `飞行记录`

第四阶段：

- 回头补空白页：`一键调飞`、`飞行控制台`

## 9. 风险与处理策略

### 风险 1：新旧组件混用导致风格不统一

处理：

- 第一阶段只改单页
- 用新页面建立明确的设计基线

### 风险 2：现有业务逻辑被改坏

处理：

- 先保留数据逻辑
- 优先替换页面结构和展示层
- 不在第一轮重做所有复杂交互

### 风险 3：样式冲突

处理：

- 新页面样式局部化
- 避免继续向旧全局 SCSS 大量追加复杂规则

### 风险 4：地图和规划类能力迁移成本高

处理：

- 第一阶段只改“壳子”
- 把复杂地图体验拆到后续阶段

## 10. 验收标准

当以下条件满足时，认为第一阶段成功：

1. `航线管理` 页面整体视觉明显转向 `shadcn-vue` 风格
2. 查询、重置、新增、编辑、删除、分页等核心功能仍可使用
3. 页面代码结构比当前版本更清晰
4. 至少沉淀出一套可复用的飞行模块页面骨架
5. 本地开发环境可正常运行
6. 构建可以通过
