# 低空平台前端对接指南

## 1. 文档说明

### 1.1 文档目标

本文面向 `E:\高栏港\dikong\fronted` 仓库的前端开发者，用于统一低空平台业务模块的对接标准，确保不同人员在接入接口、定义类型、处理字段映射、编写页面时遵循同一套规则，避免各写各的、联调成本反复增加。

本文重点解决以下问题：

- 页面字段以什么为准
- 后端 `snake_case` 字段如何在前端落地
- 请求层、API 层、页面层各自承担什么职责
- 分页、错误码、认证、多租户上下文如何统一处理
- 任务管理、航线管理、飞行记录、无人机管理四个模块分别应该展示哪些字段、删除哪些字段、保留哪些表单项

### 1.2 适用范围

本文适用于以下场景：

- `fronted` 仓库中的业务页面接口对接
- 新模块 API 文件、类型文件、字段映射函数的编写
- 现有静态原型页面向真实后端接口迁移
- 多人并行开发时的统一执行标准

### 1.3 关联文档与规则优先级

本项目当前有三类文档，各自职责如下：

1. `api设计规范v2.md`
   - 负责协议层与协作层标准
   - 包括响应结构、分页规则、错误码、认证方式、多租户上下文、命名风格目标

2. `字段.md`
   - 负责当前阶段已经确认的页面字段范围
   - 明确每个页面最终展示哪些字段、删除哪些原型字段、保留哪些表单项

3. 本文《前端对接指南》
   - 负责把上面两份文档翻译成前端开发可执行规则
   - 明确请求层、类型层、页面层应该怎么落地

当出现口径冲突时，按以下顺序处理：

1. 优先遵循 `api设计规范v2.md` 的通用规范
2. 再遵循 `字段.md` 的模块字段收口结果
3. 本文负责给出前端实现方式，不单独创造新的业务字段

## 2. 总体原则

### 2.1 前端内部统一使用 camelCase

前端内部包括以下层级，统一使用 `camelCase`：

- 页面组件
- 表单状态
- 表格数据
- store 状态
- 业务类型定义
- API 返回给页面的对象

示例：

| 后端字段 | 前端内部字段 |
|---|---|
| `route_name` | `routeName` |
| `scheduled_at` | `scheduledAt` |
| `flight_no` | `flightNo` |
| `device_sn` | `deviceSn` |
| `created_by_tenant_member_id` | `createdByTenantMemberId` |

禁止在页面、组件、store 中直接扩散 `snake_case` 字段。

### 2.2 后端 wire 字段由 API 层统一映射

如果后端当前返回的是 `snake_case`，必须在 API 层或 adapter 层完成映射。

职责分工如下：

- 请求层：统一发送请求、注入 Token 和租户头、解包响应
- API 层：负责接口路径、请求参数转换、响应字段映射
- 页面层：只消费前端内部统一后的 `camelCase` 数据

页面层禁止做以下事情：

- 直接访问 `route_name`、`created_at`、`device_sn` 这类 wire 字段
- 直接拼接 `X-TENANT-CODE`
- 直接判断 `code === "00000"`
- 直接处理后端原始包裹结构

### 2.3 以真实后端字段为准，清理静态原型字段

当前多个页面包含大量静态原型字段。凡是不在后端接口中的字段，不得继续当作真实业务字段保留。

处理原则：

- 后端已提供：页面必须展示或在表单中保留
- 后端未提供：页面不得继续展示为真实业务字段
- 后端未提供但页面内部必须使用：只能作为编辑器内部模型存在，不得伪装成后端返回字段

### 2.4 API 文件必须承担适配职责

每个业务 API 模块至少承担 3 件事：

1. 请求参数转换：`camelCase -> snake_case`
2. 响应字段转换：`snake_case -> camelCase`
3. 返回页面可直接消费的数据结构，而不是裸后端对象

## 3. 通用对接规范

### 3.1 接口基础路径

业务接口统一使用：

```http
/api/v1
```

### 3.2 统一响应结构

普通 JSON 接口统一响应结构如下：

```json
{
  "code": "00000",
  "msg": "success",
  "data": {}
}
```

前端请求层统一处理规则：

- 先判断 HTTP 状态码
- 再判断业务码 `code`
- `code === "00000"` 时返回 `data`
- 非成功业务码统一抛出异常并提示 `msg`

当前仓库中，这部分能力已经由 `src/utils/request.ts` 承担，不允许页面重复实现。

### 3.3 分页规则

分页入参统一使用：

- `pageNum`
- `pageSize`

分页响应统一消费：

- `data.list`
- `data.total`

页面层禁止直接消费：

- `count`
- `results`
- `next`
- `previous`

若后端临时返回的是 DRF 原生分页结构，也必须先在 API 层适配成项目统一的 `PageResult<T>`。

### 3.4 错误码规则

当前前端重点处理以下业务码：

| 业务码 | 含义 | 前端处理要求 |
|---|---|---|
| `00000` | 成功 | 返回 `data` |
| `A0401` | 未登录或登录失效 | 清理登录态并跳转登录 |
| `A0403` | 无权限或租户上下文失效 | 提示并按租户恢复逻辑处理 |

其他业务码处理原则：

- 显示后端返回的 `msg`
- 不在页面里做模块级自定义散装判断
- 如需专门分支处理，必须集中在 API 层或统一错误处理层

### 3.5 认证与多租户规则

#### 3.5.1 鉴权

统一使用：

```http
Authorization: Bearer <token>
```

该逻辑由 `src/utils/request.ts` 自动注入，页面和 API 模块不得重复拼接。

#### 3.5.2 多租户

业务接口需要在合适场景下注入：

```http
X-TENANT-CODE: <tenantCode>
```

规则如下：

- 业务接口：由请求层自动注入 `X-TENANT-CODE`
- IAM 全局接口：不注入 `X-TENANT-CODE`
- 页面不得手动设置租户头
- 是否带租户头由请求层统一判断，不允许各页面自由决定

### 3.6 请求层与类型层现状

当前仓库中已经存在的公共能力：

- `src/utils/request.ts`
  - 统一解包 `code/msg/data`
  - 自动注入 `Authorization`
  - 自动为业务接口注入 `X-TENANT-CODE`
  - 统一处理 `A0401`、`A0403`

- `src/types/api/common.ts`
  - `ApiResponse<T>`
  - `BaseQueryParams`
  - `PageResult<T>`

新增业务模块时必须复用这套基础设施，不允许各自重复造轮子。

## 4. 前端实现标准

### 4.1 推荐的类型分层

每个业务模块至少定义以下 3 类对象：

1. `Wire` 类型
   - 对应后端原始返回
   - 保留后端字段命名

2. `View` / `Item` 类型
   - 对应页面展示
   - 使用前端统一的 `camelCase`

3. `Form` 类型
   - 对应表单状态
   - 使用前端统一的 `camelCase`

推荐命名：

- `TaskWire`
- `TaskItem`
- `TaskForm`
- `TaskQuery`

### 4.2 推荐的映射函数

每个模块必须提供显式映射函数，不要把字段转换散落在页面里。

示例：

```ts
interface MissionWire {
  id: number;
  name: string;
  route_name: string;
  drone_name: string;
  scheduled_at: string;
  sync_status: string;
  execution_status: string;
  last_sync_at: string;
  created_at: string;
  updated_at: string;
}

interface TaskItem {
  id: number;
  name: string;
  routeName: string;
  droneName: string;
  scheduledAt: string;
  syncStatus: string;
  executionStatus: string;
  lastSyncAt: string;
  createdAt: string;
  updatedAt: string;
}

function mapMissionWireToTaskItem(wire: MissionWire): TaskItem {
  return {
    id: wire.id,
    name: wire.name,
    routeName: wire.route_name,
    droneName: wire.drone_name,
    scheduledAt: wire.scheduled_at,
    syncStatus: wire.sync_status,
    executionStatus: wire.execution_status,
    lastSyncAt: wire.last_sync_at,
    createdAt: wire.created_at,
    updatedAt: wire.updated_at,
  };
}
```

### 4.3 推荐的 API 模块模板

```ts
import request from "@/utils/request";
import type { PageResult } from "@/types/api";

interface TaskWire {
  id: number;
  name: string;
  route_name: string;
  drone_name: string;
  scheduled_at: string;
  status: number;
  sync_status: string;
  execution_status: string;
  last_sync_at: string;
  created_at: string;
  updated_at: string;
}

interface TaskItem {
  id: number;
  name: string;
  routeName: string;
  droneName: string;
  scheduledAt: string;
  status: number;
  syncStatus: string;
  executionStatus: string;
  lastSyncAt: string;
  createdAt: string;
  updatedAt: string;
}

function mapTask(wire: TaskWire): TaskItem {
  return {
    id: wire.id,
    name: wire.name,
    routeName: wire.route_name,
    droneName: wire.drone_name,
    scheduledAt: wire.scheduled_at,
    status: wire.status,
    syncStatus: wire.sync_status,
    executionStatus: wire.execution_status,
    lastSyncAt: wire.last_sync_at,
    createdAt: wire.created_at,
    updatedAt: wire.updated_at,
  };
}

const TaskAPI = {
  async getPage(params: { pageNum: number; pageSize: number }) {
    const data = await request<any, PageResult<TaskWire>>({
      url: "/api/v1/missions",
      method: "get",
      params,
    });

    return {
      list: data.list.map(mapTask),
      total: data.total,
    };
  },
};
```

### 4.4 页面禁止事项

页面、表单、store 中禁止出现以下写法：

- 直接使用后端原始字段名
- 直接 `response.data.code === "00000"`
- 直接在页面写 `headers: { "X-TENANT-CODE": ... }`
- 直接把编辑器内部草稿字段当作后端实体字段
- 继续使用已经被字段对照表明确删除的原型字段

### 5.2.2 航线编辑页内部字段

航线编辑页不是简单 CRUD 表单，而是页面内部维护一套航线草稿模型，用于编辑航线几何和配置。该模型属于前端内部编辑状态，不等于后端航线实体字段。

推荐保留的编辑页内部字段：

| 前端内部字段 | 中文名 | 说明 |
|---|---|---|
| `routeName` | 航线名称 | 编辑器内显示名称 |
| `routeType` | 航线类型 | 点状 / 面状 / 环状 |
| `department` | 所属部门 | 编辑器内部业务信息 |
| `points` | 航点集合 | 航线几何数据 |
| `globalConfig` | 全局配置 | 航线级别参数 |
| `pointConfig` | 点状航线配置 | 点状任务参数 |
| `areaConfig` | 面状航线配置 | 面状任务参数 |
| `loopConfig` | 环状航线配置 | 环状任务参数 |

说明：

- 这些字段只属于编辑器内部模型
- 它们可以存在于页面状态中
- 但不能直接当作后端航线实体字段展示在列表中

### 5.2.3 接口提交字段

用户完成航线编辑后，前端应根据当前草稿生成 XML 文件，并将其提交给后端处理。

最终提交字段如下：

| 后端字段 | 前端来源 | 中文名 | 说明 |
|---|---|---|---|
| `name` | `routeName` | 航线名称 | 提交给后端的航线名称 |
| `xml_file` | 前端生成的 XML 文件 | 航线XML文件 | 由编辑页生成，不是用户手工上传 |

### 5.2.4 航线模块标准数据流

航线模块必须遵循以下流程：

1. 列表页读取后端航线记录
2. 编辑页维护内部航线草稿模型
3. 用户保存或提交时，先校验编辑器草稿
4. 由前端根据草稿生成 XML 文件
5. 调用后端航线处理接口，提交 `name + xml_file`
6. 接口成功后刷新航线记录列表或回写当前记录状态

### 5.2.5 航线模块禁止事项

明确禁止以下写法：

- 把 `xml_file` 写成用户手工上传控件
- 把 `points`、`globalConfig` 等内部字段直接当后端列表字段展示
- 把编辑器内部模型原样直接当接口提交对象
- 把当前仓库中的临时 KMZ / 外部 wayline 下发流程当作最终业务标准
